ROOT CAUSE SUMMARY:
  The register allocator in the tc compiler is assigning RSP and RBP as
  general-purpose temporary/spill registers. These registers are SPECIAL-PURPOSE
  in x86-64 ABI and MUST be excluded from the allocatable register pool.
  RSP = Stack Pointer (always points to top of stack)
  RBP = Base Pointer (frame pointer, conventionally preserved)

--- BUG ANALYSIS: compare_branch ---

BUG #1: mov rsp, 3 (at 0x401056)
  - The compiler allocated the constant value "3" to the RSP register.
  - RSP is the stack pointer. Writing to it destroys the stack frame.
  - This causes immediate SIGSEGV when subsequent instructions try to
    access the stack (push/pop/epilogue).
  - Should have used a callee-saved register (r12-r15) or a volatile register
    (rax, rcx, rdx, rsi, rdi, r8-r11).

BUG #2: cmp rbp, rsp (at 0x40105d)
  - Due to BUG #1, RSP = 3. RBP = frame pointer.
  - This compares the frame pointer address with the constant 3.
  - The comparison is meaningless (intended: compare variable x with constant 3).
  - The comparison result is garbage, so the branch decision is random.

BUG #3: lea rsp, [rbp - 4] (at 0x401077, 0x401089)
  - The compiler uses RSP to hold the address of local variable x.
  - This overwrites the stack pointer with a local variable address.
  - The epilogue's "mov rsp, rbp" tries to recover, but by then the stack
    is already corrupted.

BUG #4: mov rbp, 0xa (at 0x401070)
  - The compiler overwrites RBP (frame pointer) with the constant 10.
  - This destroys the frame pointer, making the epilogue unable to
    restore the stack or return address.

BUG #5: Return value uses wrong register
  - The function returns via "mov rax, rdi" at 0x401096.
  - rdi was set to 5 at the beginning (the initial value of x).
  - It should return the current value of x (10 if branch taken).
  - This is a secondary bug related to register allocation for the return value.

--- RUNTIME RESULT ---
  Exit code: -11 (SIGSEGV)
  Cause: RSP corrupted by "mov rsp, 3" at 0x401056.
  The process crashes when trying to access the corrupted stack.

================================================================================
CROSS-CUTTING ANALYSIS
================================================================================

1. RSP/RBP MISUSE (PRIMARY ROOT CAUSE)
   ====================================
   The register allocator treats RSP and RBP as allocatable registers.
   They are assigned to hold:
   - Constant values (mov rsp, 3; mov rbp, 2)
   - Computed addresses (lea rsp, [rbp - 4])
   - Boolean results (movzx rsp, al)
   - Arithmetic results (add rbp, rsi)
   
   FIX: Exclude RSP and RBP from the virtual register allocation pool.
   Only RAX, RCX, RDX, RSI, RDI, R8-R15 should be allocatable.
   RSP is managed by the prologue/epilogue and call convention.
   RBP is managed by the prologue/epilogue.

2. BRANCH DISPLACEMENTS
   =====================
   The branch targets appear correct in the disassembly (je, jmp, call
   all target valid addresses within the .text section). However, the
   semantic correctness of branches is wrong because the compared values
   are wrong (due to register misuse).

3. STACK ALIGNMENT BEFORE CALLS
   ==============================
   In function_call, main() pushes 11 registers before the call to double().
   11 * 8 = 88 bytes. The entry stub does "and rsp, ~15; sub rsp, 8".
   After pushing rbp (+8), the stack is at rsp-8. Then 10 more pushes = 80.
   Total from entry: 8 + 8 + 80 = 96. 96 % 16 = 0. So alignment is OK.
   However, the call instruction itself pushes an 8-byte return address,
   making the callee's stack 8 bytes off from 16-byte alignment.
   The entry stub's "sub rsp, 8" compensates for this. So alignment is
   technically correct, but fragile.

4. INVALID CALL TARGETS
   =====================
   All call targets (0x401022, 0x40105c) point to valid code within
   the .text section. No invalid call targets detected.

5. SECONDARY ISSUES
   =================
   a) Uninitialized register stores: r15d used before initialization.
   b) Wrong register for return value: rdi used instead of computed result.
   c) Excessive register saving: all caller-saved regs pushed before calls.
   d) Variable address computation corrupts frame pointer.

================================================================================
RECOMMENDATION
================================================================================

CRITICAL FIX: In the register allocator (src/x86_64.c or src/x86_64_instr.c):
  1. Remove RSP and RBP from the set of allocatable registers.
  2. The allocatable register pool should be:
     - Caller-saved (volatile): RAX, RCX, RDX, RSI, RDI, R8, R9, R10, R11
     - Callee-saved (non-volatile): RBX, R12, R13, R14, R15
  3. RSP is managed implicitly by the compiler (prologue/epilogue/sub rsp).
  4. RBP is managed explicitly as the frame pointer (push rbp/mov rbp,rsp/.../pop rbp).
  5. Variable addresses should use [RBP - offset], not store addresses in RSP.
